# Phase 7.5 — Profile pictures (avatars)

Users can upload, replace, and remove a profile picture from their own profile page. The picture surfaces on every social surface where a user is named: the profile header, the friends list (all three tabs and the user-search dropdown), the invites page (incoming + outgoing rows), and the Listen Later participant chips. Storage lives behind a small `Storage` Protocol with three implementations (`Local`, `R2`, `InMemory` for tests) so the same code path works locally and in production on Render + Cloudflare R2.

## Notes / decisions

- **Why R2 over Postgres-bytea**: Render's filesystem is ephemeral, so local-only is a non-starter. Postgres-bytea works but every avatar render would have to round-trip through FastAPI; R2's public bucket lets the browser hit the CDN directly. The protocol-based abstraction means we keep `local` in dev so contributors don't need an R2 account.
- **Why batched URL lookups instead of embedding User objects**: friendship + invite payloads already serialise their referenced users by username. Embedding the whole user record would mean schema churn across every consumer; one `picture_url` field per username slot keeps the existing shape and the resolver code is a single `WHERE username IN (...)`.
- **Cache-busting via the key, not via query strings**: every replace generates a new `{shortuuid}` segment, so `<img src>` URLs change deterministically and we don't need `?v=…` busters. The old object is `DELETE`'d after the new key is committed.
- **Why no server-side resize (Pillow)**: avoided a new library this phase. Combined with the 2 MB cap at the upload boundary, R2 objects stay small enough for avatar use. The README will eventually note this if we hit it.
- **Test storage backend**: `InMemoryStorage` exposes its `objects: dict[str, bytes]` so tests can assert exactly what was uploaded and that replace really deletes the prior key. Reused across `test_avatar.py` and the existing friendship/invite tests via the autouse `storage` fixture in `conftest.py`.
- **Frontend `<input type="file">` quirk**: `e.target.value = ""` after reading the file so the user can pick the same filename twice in a row (browsers otherwise treat the second `change` as a no-op).
