# Phase 5 — Friends

Adds friend requests, friend search, and enforces friends-only visibility on private profile dashboards.

## Files

### Backend

- [backend/app/models/friendship.py](../../backend/app/models/friendship.py) — new `Friendship` model. Stores a canonical ordered pair (`user_a_username < user_b_username`) with a `UniqueConstraint` + `CheckConstraint`, so each pair has at most one row regardless of who initiated. Tracks `status` (`pending`/`accepted`), `requested_by`, `created_at`, `accepted_at`.
- [backend/app/models/\_\_init\_\_.py](../../backend/app/models/__init__.py) — registered `Friendship`/`FriendshipStatus`.
- [backend/alembic/versions/c08ff9961194_add_friendships.py](../../backend/alembic/versions/c08ff9961194_add_friendships.py) — autogenerated migration for the new table + indexes.
- [backend/app/services/friendship.py](../../backend/app/services/friendship.py) — `ordered_pair(a, b)`, `get_friendship(db, u1, u2)`, and `are_friends(db, u1, u2)`. The single source of truth for the canonical lookup that the router and the dashboard visibility check both use.
- [backend/app/schemas/friendship.py](../../backend/app/schemas/friendship.py) — `FriendshipCreate`, `FriendshipResponse`, `FriendshipListResponse` (buckets: `incoming`/`outgoing`/`accepted`), `UserSearchResult`.
- [backend/app/routers/friendships.py](../../backend/app/routers/friendships.py) — `POST /friendships` (send), `POST /friendships/{id}/accept`, `POST /friendships/{id}/decline` (deletes the pending row so the pair can re-request later), `DELETE /friendships/{id}` (unfriend or cancel), `GET /friendships/me`. Endpoints reject self-friending, duplicates (from either direction), accept/decline of your own request, and any access by a user who isn't in the pair.
- [backend/app/routers/users.py](../../backend/app/routers/users.py) — added `GET /users/search?q=…&limit=10` (matches `username` or `display_name`, excludes self, case-insensitive `LIKE`). Wired `are_friends` into the dashboard visibility check so a `private` profile is now also visible to accepted friends, not just the owner.
- [backend/app/main.py](../../backend/app/main.py) — registered the friendships router.
- [backend/tests/test_friendships.py](../../backend/tests/test_friendships.py) — 15 tests covering: send happy path, no self-friend, no unknown user, no duplicates (same direction and reverse direction), accept transitions to accepted, no accepting your own request, decline removes the row, unfriend removes accepted, strangers get 403 on accept/delete, `/friendships/me` bucket grouping, user search matches both fields and excludes self, private profile visible to a friend but blocked for a pending request.

### Frontend

- [frontend/src/api/friendships.ts](../../frontend/src/api/friendships.ts) — typed wrappers around the five friendship endpoints + `searchUsers`.
- [frontend/src/pages/FriendsPage.tsx](../../frontend/src/pages/FriendsPage.tsx) — single page with a top search bar (form submits to `/users/search`, results render inline with an "Add friend" button) and three tabs (Friends / Incoming / Outgoing). Each card links to the user's profile; accepted friends also link to their dashboard. Errors from `POST /friendships` are surfaced from the API `detail`.
- [frontend/src/pages/FriendsPage.module.css](../../frontend/src/pages/FriendsPage.module.css) — tab + card styling.
- [frontend/src/App.tsx](../../frontend/src/App.tsx) — added `/friends` route gated by `ProtectedRoute`.
- [frontend/src/components/NavBar.tsx](../../frontend/src/components/NavBar.tsx) — added a "Friends" nav link.

## Notes / decisions

- **Username-keyed pair instead of `user_id`**: the plan says `user_a_id`/`user_b_id`, but Phase 2's migration moved foreign keys to `users.username` for ratings, and the dashboard route already keys off usernames. Keeping friendship username-keyed avoids a join and matches the rest of the schema. Sort order is lexicographic, which is fine since usernames are immutable in this app.
- **Search modal vs. inline search**: the plan said "search modal"; chose inline search on the same page because it keeps the friends-list refresh logic in one component and the page is already short. No modal library was needed.
- **Decline deletes the row**: alternative considered was a third `declined` status. Deletion is simpler and lets the same pair re-request later; nothing in v1 needs the historical record.
- **No TanStack Query yet**: still `useEffect` + `useState` with a `refresh()` helper called after each mutation. Consistent with the rest of the project.
