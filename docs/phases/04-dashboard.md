# Phase 4 — Personal profile dashboard (you vs Spotify)

Replicates the demo's dashboard for a single user: lists every published rating with score, similarity vs Spotify's top-5 popular tracks, and a Chart.js line plot over time. Adds an album detail page that shows the viewer-user's top 5 next to Spotify's top 5.

## Files

### Backend

- [backend/app/models/album.py](../../backend/app/models/album.py) — added `spotify_top5_indices: list | None` JSON column on `Album` for caching Spotify's top 5 popularity ranks (1-based, matching `AlbumTrack.index`).
- [backend/alembic/versions/4ee1b42d7212_add_spotify_top5_to_albums.py](../../backend/alembic/versions/4ee1b42d7212_add_spotify_top5_to_albums.py) — autogenerated migration adding the new column.
- [backend/app/services/spotify.py](../../backend/app/services/spotify.py) — added `get_top5_popular_indices(spotify_id)` which fetches per-track popularity in batches of 50 and returns the 5 highest-ranked track numbers.
- [backend/app/schemas/dashboard.py](../../backend/app/schemas/dashboard.py) — new `DashboardAlbum`, `DashboardEntry`, `DashboardResponse` Pydantic models.
- [backend/app/routers/users.py](../../backend/app/routers/users.py) — added `GET /users/{username}/dashboard?compare_to=spotify`. Returns all published ratings; computes similarity against the cached Spotify top 5 using `compute_ranking_loss` + `compute_similarity_score` and the `BaselineStat` row for the album's track count. Enforces visibility: `public` is visible to anyone authenticated; `private` is only visible to the owner (friend check deferred to Phase 5).
- [backend/tests/test_dashboard.py](../../backend/tests/test_dashboard.py) — 9 tests covering happy path, similarity math, draft exclusion, top-5 caching, public/private visibility, 404, and unauthenticated access.

## Notes / decisions

- **Top-5 indexing**: `AlbumTrack.index` is 1-based (Spotify's `track_number`). To make `compute_ranking_loss` work, `spotify_top5_indices` is stored in the same 1-based convention rather than the reference's 0-based one.
- **Spotify call site**: top-5 popularity is computed lazily on first dashboard request per album and cached on `Album.spotify_top5_indices`. The test `test_dashboard_caches_spotify_top5` verifies the Spotify client is only called once across two dashboard fetches.
- **Date range control**: used two HTML `<input type="date">` fields instead of `nouislider` to avoid a new dependency. The phase didn't strictly need a slider — two date pickers cover the same UX.
- **TanStack Query**: still not introduced. Existing pages use plain `useEffect` + `useState`, and the dashboard is a single fetch — Query would be premature here.
